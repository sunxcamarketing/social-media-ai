# Plan: Full i18n (German / English) — Platform-Wide

**Created:** 2026-04-19
**Status:** Draft
**Request:** Add a language toggle (DE/EN) that translates the entire platform, allows selecting a language per client at onboarding, and drives Chat + Voice agents + all content generation to produce output in the client's language.

---

## Overview

### What This Plan Accomplishes

Makes SUNXCA fully bilingual (German ↔ English). Every client gets a language assigned at onboarding — strategies, scripts, audits, Chat Agent replies, and Voice Agent speech are generated in that language. Admin + Portal UI can be toggled on the fly. Existing German clients continue working unchanged.

### Why This Matters

Unlocks the US / UK / international market for Aysun's agency. Without this, English-speaking clients get German UI, German-speaking Voice Agent, and German scripts — the platform is unusable for them. Expands TAM beyond DACH.

### Gaps or Problems Being Addressed

1. **UI toggle only stores user preference** (localStorage) — not tied to client. A German client's portal will display in whatever language the admin last chose.
2. **No `configs.language`** — pipelines cannot branch on client language.
3. **Onboarding has no language selection** — all new clients implicitly German.
4. **100% of prompts are German** — no English output possible regardless of config.
5. **Chat Agent / Voice Agent** output language-agnostic — they'd speak German even to English clients.
6. **Client Portal** (`src/app/portal/*`) uses **zero** `useI18n` calls — all hardcoded German strings.
7. **Voice Agent greeting trigger** is a hardcoded German sentence in `voice-server.ts`.
8. **`verboten-ai-sprache.md`** lists 100+ **German** AI phrases — no English equivalents exist.

## Design Decisions

### Key Decisions Made

1. **Keep the existing lightweight i18n — do NOT install next-intl.**
   - Rationale: 611 lines of working code with 500 translations already exist. Next-intl would require file restructure (`messages/` dir), middleware reconfig, and Provider rewrite. 5-10× more work for ~zero additional capability at this scale. The current system handles placeholders (`{{count}}`), toggles, and localStorage already.
2. **Prompt filename convention: `.md` = German default, `.en.md` = English variant.**
   - Rationale: All 44 existing files keep their names. Zero risk of breaking existing `buildPrompt("content-agent")` calls. English is opt-in via explicit `lang: "en"`. Fallback chain: `{name}.en.md` → `{name}.md` (so missing English files don't crash — they just fall back to German with a console warning).
3. **Language precedence:** user-override (localStorage) > `config.language` > `"de"`.
   - Rationale: Aysun (admin) should be able to override UI language locally without affecting what her clients see. Clients see their assigned language by default but can override for their own session.
4. **Voice Agent: accept `lang` via WS query param AND fallback to `config.language`.**
   - Rationale: Query param lets the frontend override (e.g., admin testing in EN); config default applies for normal clients.
5. **English `verboten-ai-sprache.en.md` must be hand-curated.**
   - Rationale: English AI tells differ entirely from German ("delve", "tapestry", "game-changer", "In today's fast-paced world", "It's important to note"). Direct translation of the German list produces nonsense. This is the single most important prompt module.
6. **Onboarding language step is FIRST** (step 0 before "basics").
   - Rationale: Changing language mid-flow would require re-translating all subsequent placeholders (already in German). Asking first means the rest of the wizard could even be presented in EN for English clients (optional enhancement).
7. **Backward compatibility:** existing clients without `language` → default `"de"` via column default. No data migration needed.
8. **`buildPrompt()` signature stays additive:** `buildPrompt(name, subs = {}, lang?: Lang)`. All ~30 existing call sites keep working during rollout; we update them incrementally.

### Alternatives Considered

- **next-intl with `messages/de.json` / `messages/en.json`:** rejected — see Decision 1. Can be revisited in a future refactor if translations grow past ~2000 keys.
- **Inline conditional prompts** (`{{de:Text}}{{en:Text}}` syntax): rejected — turns prompt files into template soup; kills the "readable markdown" property the modular system was built for.
- **Auto-translate prompts at request-time** (Claude translates DE → EN on the fly): rejected — adds latency + cost per request, breaks prompt caching, introduces translation drift. Offline curation is strictly better.
- **Storing `preferred_language` on `client_users`** (per-user override): rejected for v1 — localStorage already gives per-user override, and most clients only have one user each. Can add later if needed.

  export function I18nProvider({ children, initialLang }: { children: ReactNode; initialLang?: Lang }) {
